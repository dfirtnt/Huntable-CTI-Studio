SIGMA Similarity Engine

How Huntable CTI Studio determines whether a proposed detection rule overlaps with existing SigmaHQ coverage

Pipeline Overview

When you click Similarity Search on a queued rule, this pipeline runs:

1
Parse & Canonicalize
The queued rule's YAML is parsed. Logsource is normalized to a key like windows|process_creation. If the sigma_similarity package is installed, the rule is decomposed into a canonical class (e.g. windows.process_creation) and precomputed atom sets.
2
Retrieve Candidates
Deterministic mode: all SigmaHQ rules with the same canonical_class (no limit). Legacy mode: rules matching logsource_key, limited to top 20. This is a hard gate — rules from a different log source are never compared.
3
Compute Atom Jaccard
Each candidate is compared atom-by-atom. Atoms are the irreducible predicates — e.g. Image|endswith|\rundll32.exe. The Jaccard index measures set overlap. New: cross-field soft matching catches rules targeting the same executable via different fields.
4
Score & Rank
Final similarity = 0.70 × atom_jaccard + 0.30 × logic_shape − penalties. Penalties apply for service mismatches and filter divergence. Top 20 results are returned; those with jaccard > 0 appear in the UI.
5
Classify Novelty
DUPLICATE jaccard > 0.95 + logic > 0.95   SIMILAR jaccard > 0.80   NOVEL everything else

What Is an Atom?

An atom is one irreducible behavioral constraint from a SIGMA rule's detection block. It captures exactly one field-operator-value check.

field | operator | op_type | value

SIGMA YAML

detection: selection: ParentImage|endswith: '\services.exe' Image|endswith: '\rundll32.exe' condition: selection

Extracted Atoms

ParentImage|endswith|literal|\services.exe
Image|endswith|literal|\rundll32.exe

Atom Jaccard — Strict vs Soft Matching

Jaccard index = |A ∩ B| / |A ∪ B|. Two atoms match if their entire key is identical (field + operator + value). This is the strict match.

How soft matching works: When strict jaccard = 0, the engine extracts executable values from process-related fields (Image, CommandLine, ParentImage, etc.) and checks for shared values. Same exe, different field → partial credit at 50% weight.

shared_exe_values = {\rundll32.exe} → 1 match
union_atoms = 5 total unique atoms (2 proposed + 3 SigmaHQ)

soft_jaccard = (1 / 5) × 0.5 = 0.10 // deterministic path (precomputed atoms)
soft_jaccard = (1 / 6) × 0.5 = 0.083 // legacy path (includes CommandLine:'' atom)

Process Executable Field Group

Soft matching only applies to fields that reference executables/binaries. These fields form a semantic group — the same \rundll32.exe value in any of them means "this rule is about rundll32":

Image ParentImage CommandLine ParentCommandLine OriginalFileName process.image process.parent_image process.command_line process.parent_command_line process.original_file_name

Blue = legacy atom names (from YAML field parsing)   Purple = canonical names (from sigma_similarity deterministic engine)

Fields not in this group (e.g. dns.query, TargetFilename, registry.path) are never soft-matched. A DNS rule and a process rule won't get false overlap.

Deterministic vs Legacy Path

The engine has two comparison modes depending on whether sigma_similarity is installed:

Deterministic when sigma_similarity installed
  • Atoms precomputed at index time, stored in DB
  • Fields normalized to canonical process.* namespace
  • No YAML re-parsing at query time — pure set math
  • Candidates filtered by canonical_class, no limit
  • Uses compute_containment() for logic scoring
  • Soft match via _soft_exe_jaccard_from_atom_strings()
Legacy fallback when package unavailable
  • Atoms extracted at query time from YAML
  • Fields title-cased (e.g. Image, CommandLine)
  • Full YAML parsing per candidate
  • Candidates filtered by logsource_key, limit 20
  • Uses compute_logic_shape_similarity()
  • Soft match via _exe_values() inner function

Where It Runs

Everything runs in the main app container (Dockerfile, Python 3.11 slim). There is no separate similarity service.

// sigma_semantic_similarity is a local workspace package pyproject.toml → members = ["sigma_semantic_similarity"] Dockerfile:71 → COPY sigma_semantic_similarity/ ./sigma_semantic_similarity/ Dockerfile:72 → uv sync --frozen // installs it as editable dep

The deterministic path is only active inside Docker (where the package is installed). Your local dev shell uses the legacy path since sigma_similarity isn't in the local venv.
